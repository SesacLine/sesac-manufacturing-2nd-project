# API 명세서 — 변경 이력 (CHANGELOG)

> `API_명세서_v1.0.md` 본문에서 제거한 **과거 변경 이력·설계 논의 과정**을 모은 문서다. v1.0 본문은 "현재 유효한 스펙"만 담고, "이전엔 X였는데 Y로 바꿨다 / 어떤 대안을 검토하다 폐기했다" 류 서술은 전부 이리로 옮겼다.
>
> 각 항목 포맷:
> ```
> ### [필드/엔드포인트명]
> - 변경 시점:
> - 이전 상태:
> - 변경 후 상태:
> - 변경 이유:
> ```
> 원문에 이유가 명확히 안 적힌 항목은 "이유 미기재"로 표시했다.

## A. 기획안 v1.4 반영 (원문 "변경 이력 v0.1 → v0.2" 절)

### [`tier` (검증등급) 필드]
- 변경 시점: v0.1 → v0.2
- 이전 상태: 원인 후보에 검증등급 필드 없음
- 변경 후 상태: 원인 후보(2.5 가설 객체·2.7 근거)마다 `tier` 필드 신설. 값 `auto`(자동, 데이터로 즉시 채택/기각) / `semi_auto`(반자동, 사람 판정 필요) / `none`(근거없음, 문헌 서술만)
- 변경 이유: 기획안이 "후보마다 검증등급에 따라 MCP 도구를 조건부로 호출"하도록 바뀜. 프론트는 이 값으로 배지/색을 구분 표시

### [가설 판정 `adopted` → `verdict`]
- 변경 시점: v0.1 → v0.2
- 이전 상태: 가설 판정이 `adopted`(참/거짓) 두 값
- 변경 후 상태: `verdict`(3-state) — `accepted`(채택) / `rejected`(기각) / `insufficient`(근거부족). 비채택 사유는 `verdict_reason`에 담고 `accepted`이면 `null`. (위치: 2.5 가설 객체)
- 변경 이유: 참/거짓 두 값으로는 "왜 기각됐는지"를 담지 못함. 기획안 Critic 노드가 기각 사유(시간 선후 뒤집힘, 정상 로트 대조 미수행 등)를 남기므로 이를 그대로 전달

### [분석 상태 `status`에 `insufficient` 추가]
- 변경 시점: v0.1 → v0.2
- 이전 상태: 결과 없음을 `unmapped` 하나로 뭉쳐 표현
- 변경 후 상태: `reviewed`(채택된 가설 있음) / `insufficient`(신규 — 원인 매핑은 됐으나 Critic이 채택 가능한 후보를 못 찾음, 판단 불가) / `unmapped`(패턴 자체가 원인 매핑 대상 아님)으로 분리. (위치: 2.2 status 정의, 2.5 응답 3종)
- 변경 이유: 기획안이 "근거부족(insufficient_evidence)"과 "매핑없음"을 명확히 구분하며, 프론트도 두 경우를 다르게 안내해야 함

### [권장 조치 `next_actions` 필드]
- 변경 시점: v0.1 → v0.2
- 이전 상태: 권장 조치 필드 없음
- 변경 후 상태: 채택된 원인에 대한 후속 조치 리스트 `next_actions` 신설. (위치: 2.5 가설별, 2.7 근거)
- 변경 이유: 기획안 기능 5가 "Root Cause **및 권장 조치** 생성"으로, Hypothesis 워크플로우 반환값에 `next_actions`가 포함됨

### [배치 입력 방식 (`POST /batches` 본문)]
- 변경 시점: v0.1 → v0.2 (기획안 v1.4에서 확정)
- 이전 상태: 분석 트리거/입력 방식 미확정 (자유 질의 가능성 열려 있었음)
- 변경 후 상태: 자유 질의 없음 → 고정 질문 템플릿 + 하루 1회 배치 트리거로 확정. `POST /batches` 요청 본문 `{}`(입력 없음). (위치: 2.3 입력 설명)
- 변경 이유: 기획안이 "자유 질의 없음 → 고정 질문 템플릿 + 하루 1회 배치 트리거"로 확정. 이로써 요청 본문이 `{}`인 근거가 명확해짐

### [고정 질문 문자열(`question`)을 응답 필드로 싣는 안]
- 변경 시점: v0.1 → v0.2 (검토 후 미채택)
- 이전 상태: 고정 질문 문자열(`question`)을 응답 필드로 싣는 안 검토
- 변경 후 상태: 응답 필드로 넣지 않음
- 변경 이유: UI가 쓰지 않고 `pattern`에서 결정적으로 파생 가능

### [근거 응답(2.7) 구조 정렬 + enum 명문화]
- 변경 시점: v0.1 → v0.2
- 이전 상태: 2.7 근거 응답이 기획안 Evidence 테이블 컬럼과 정렬돼 있지 않음. `status`/`tier`/`verdict`/`stage` 허용값 집합이 문서상 미명시
- 변경 후 상태: 2.7 근거 응답을 Evidence 테이블 컬럼(tier·지지·반대·미검증 인용·권장조치)에 맞춰 `tier`, `verdict`, `unverified`(검증 제외된 인용), `next_actions` 추가. `status`/`tier`/`verdict`/`stage`(공정 6종) 4개 필드 허용값 집합을 3장에 enum으로 명문화
- 변경 이유: 기획안 Evidence 테이블과의 정렬 및 "임의 문자열이 아니라 정해진 값만 온다"는 계약 명시

### [채택 확률 `adopt_prob` / `prob` 필드 제거]
- 변경 시점: v0.1 → v0.2 (기획안 v1.4 채택 모델 정리)
- 이전 상태: 대기열(2.2)에 「채택 확률」 열/`adopt_prob`, 가설(2.5)에 `prob` 등 숫자 확률 필드 존재 (예: `82%`)
- 변경 후 상태: 숫자 확률 필드 제거. 채택 신호는 `verdict`(2.5)·`status`(2.2) 범주형 판정 단독. `top_cause`는 ⑥응답생성이 지정한 대표 원인(= `hypotheses[0]`)을 API가 전달만 함. (위치: 2.2·2.5·3장 데이터 모델)
- 변경 이유: v1.4에서 채택은 Critic의 범주형 판정이고 숫자 확률을 산출하는 노드가 파이프라인에 없음. `82%` 등은 v2 와이어프레임 UI 잔재로 출처가 없고 `verdict`/`status`와 의미가 중복 (`adopted`→`verdict` 마이그레이션의 완결)

## B. 본문 내 인라인 설계 변경

### [`stage` 파생 소스]
- 변경 시점: 미상 (초안 → 현행)
- 이전 상태: `equipment → group_of` 경로로 공정(stage)을 파생 (`secsgem-mcp/simulator/fab_model.py` 정규화)
- 변경 후 상태: `hypotheses[0].stage`(KG `ProcessStep` 6종) 직접 매핑으로 확정. `stage=null`은 "공정 미상" fallback. (2.5·부록 3.2)
- 변경 이유: (1) `equipment → group_of`는 크로스-컴포넌트 의존, (2) 카드 `stage`와 description이 어긋날 여지. `stage`는 이미 KG `ProcessStep` 6종이라 정규화가 불필요하고 카드와 단일 출처를 이룸

### [`wafer_id` 형식 · die-map 경로]
- 변경 시점: 미상 (②-1 결정)
- 이전 상태: (A안) `wafer_id`를 전역 유니크 id로 두고 die-map을 전역 라우트(`/wafers/{wafer_id}`)로 제공
- 변경 후 상태: (B안 채택) `wafer_id`=로트-로컬 정수 문자열(`"1"`~`"25"`, 전역 유니크 아님). die-map은 로트 종속 경로 `/api/v1/lots/{lot_id}/wafers/{wafer_id}/die-map`
- 변경 이유: `wafer_id`가 로트-로컬이라 로트마다 `"1"`이 존재 → 전역 유니크 id 안(A)은 성립하지 않음

### [웨이퍼맵 전달 방식]
- 변경 시점: 미상 (검토 후 확정)
- 이전 상태: 좌표 배열(JSON)로 줄지 이미지로 줄지 두 안 검토
- 변경 후 상태: 이미지 방식 채택(2.6 `die_map_url`, 2.6.1 `image/png`). 좌표 JSON 안은 명시적 die 인터랙션 요구가 생기면 재검토(보류)
- 변경 이유: 현재 코드 정합상 — 원천 `die_map`은 `0/1/2` 격자 BLOB이고 MCP `get_wafer_map`이 이미 PNG(base64)로 렌더하므로 얇은 재서빙 엔드포인트만 신설하면 됨. 좌표 리스트 변환·노출 도구가 코드에 없음

### [`wafer.die_map` 용어 표기]
- 변경 시점: 미상 (정정)
- 이전 상태: "`wafer.die_map` = 이미지 BLOB"으로 표기
- 변경 후 상태: `wafer.die_map`은 `0/1/2`(배경/통과/불량) 2D 격자 배열을 `np.save`한 BLOB. 이미지는 이 격자를 렌더한 산출물이지 `die_map` 그 자체가 아님
- 변경 이유: 원천 데이터 구조(`secsgem-mcp/server/db.py`, `simulator/generate.py`) 확인 결과 이전 표기가 부정확

### [웨이퍼 수 필드 위치/구성]
- 변경 시점: 미상 (②-3 / 웨이퍼 수 필드 결정)
- 이전 상태: 검토한 세 안 — (A) 아무 필드도 없이 프론트가 `wafers.length`로 파생 / (B) `wafer_count`만 추가 / (C) `wafer_count` + `defect_count`/`normal_count`. 별도 `is_normal` 플래그 안도 검토. 2.5(분석 상세)에도 그룹 총합 `wafer_count`가 있었음
- 변경 후 상태: (C안 채택) 2.6에 `wafer_count`+`defect_count`+`normal_count` 3개 서버 집계 필드. 정상/불량 판별은 `defect_pattern == "Normal"`로 계산(`is_normal` 필드 없음). 2.5의 그룹 총합 `wafer_count`는 제거
- 변경 이유: 화면3 헤더 "판독 N장 중 M장 불량"이 정상/불량 분리 수치를 씀 → 서버 집계로 바로 렌더(C). A는 프론트가 매번 순회 집계해야 하고, B는 `wafers.length` 중복. 2.5 `wafer_count`는 소비처가 없어 제거

### [2.7 드릴다운 키 `cause` → `hypothesis_id`]
- 변경 시점: v0.2 → v0.3
- 이전 상태: 근거 상세(2.7) 조회 키로 `cause`를 path 파라미터에 사용
- 변경 후 상태: `hypothesis_id`(형식 `h{n}`, 그룹 내 가설 배열 인덱스 0-based)를 path 키로 사용. `cause`는 응답 body에 표시용으로만 잔류(조회 키 아님)
- 변경 이유: `cause`는 kg_rca가 LLM으로 자유추출한 문자열이라 (ⓐ 공백·슬래시·유니코드로 URL이 깨지고 ⓑ 같은 cause가 서로 다른 path로 중복될 수 있어) path 키로 부적합

### [`normal_ratio.interpretation` enum]
- 변경 시점: 미상 (폐기)
- 이전 상태: `normal_ratio`에 지지/약화 판단을 담는 `interpretation` enum 필드
- 변경 후 상태: `interpretation` enum 폐기. 지지/약화 판단은 `caption` 문장으로만 전달(원값 `value` / 표시 문자열 분리)
- 변경 이유: 임계값 미정이고 `verdict`와 의미 중복

### [③ GraphRAG 노드 성격 서술]
- 변경 시점: 미상 (v1.4 재설계)
- 이전 상태: ③ GraphRAG를 런타임 그래프 "검색"으로 서술 (진행 단계 표기 포함)
- 변경 후 상태: ③은 빌드타임에 결정적 순회를 끝낸 결과를 "조회"하는 것으로 서술. 결정적 룰베이스 노드
- 변경 이유: v1.4에서 GraphRAG가 빌드타임 결정적 순회로 재설계됨 — 런타임엔 사전 계산된 원인 후보를 꺼내 볼 뿐

### [비채택 사유 필드명 통일 · `verdict` 3-state 합성 규칙]
- 변경 시점: 미상 (②-D 결정)
- 이전 상태: 2.7이 비채택 사유 필드로 `reject_reason`을 사용. `verdict` 3-state 합성 방식 미확정
- 변경 후 상태: 비채택 사유 필드명을 2.5·2.7 공통 `verdict_reason`으로 통일(`reject_reason` 폐기). `insufficient`는 API가 Critic의 `rejected` 중 **고정 사유토큰**(예 `P5_NO_KG_MECHANISM`)인 항목만 승격(`verdict_reason` 자유서술 본문 매칭 금지). `verdict`(가설)·`tier`(가설)·`status`(분석)는 독립 축이라 셋 다 유지
- 변경 이유: 필드명 일관성 확보. 토큰 기반 승격은 Critic이 자연어 문구를 바꿔도 배지가 안 깨지게 하기 위함

## C. 4장 미결정 사항 중 확정(`[x]`)된 결정

### [Base URL / 버전 prefix]
- 변경 시점: 미상
- 이전 상태: 미확정
- 변경 후 상태: `http://localhost:8000/api/v1`로 확정 (1장)
- 변경 이유: 이유 미기재

### [인증 방식 유무]
- 변경 시점: 미상
- 이전 상태: 미확정
- 변경 후 상태: 확정(본문에 인증 관련 규약 없음 = 인증 없음)
- 변경 이유: 이유 미기재

### [`analysis_id` 유니크 기준]
- 변경 시점: 미상
- 이전 상태: 유니크 기준 미확정 (날짜만으로는 같은 날 2회차가 1회차를 덮어써 누적이 깨짐)
- 변경 후 상태: 배치 실행 + 패턴 단위로 확정(형식 `grp_{패턴}_{배치날짜}_{순번}`). `{배치날짜}_{순번}`으로 `batch_id` 복원 가능하므로 2.5 응답엔 `batch_id`를 싣지 않음
- 변경 이유: 배치별 결과 누적을 위함 (배치가 반복 실행돼도 과거 결과가 덮어써지지 않게)

### [배치 진행 조회 방식]
- 변경 시점: 미상
- 이전 상태: 조회 방식 미확정
- 변경 후 상태: 폴링으로 확정(반복 GET `/batches/{id}`). SSE 스트리밍(`/batches/{id}/stream`)은 로그 실시간성이 필요해지면 추가하는 향후 선택지로 오픈
- 변경 이유: 하루 1회·소수 사용자·데모 안정성. 실패도 `200+status:"failed"`로 주는 폴링 전제 설계

### [배치 진행 `steps` 노출 범위]
- 변경 시점: 미상
- 이전 상태: 노출 범위 미확정
- 변경 후 상태: 기획안 v1.4 §7.1 7노드 전체(⓪~⑥)로 확정. ③은 빌드타임 순회 결과 "조회", `logs`는 tier별 조건부 MCP 호출 트레이스 (2.4)
- 변경 이유: 이유 미기재

### [2.6 로트당 웨이퍼 장수 — "25장 고정" 서술 정정]
- 변경 시점: 2026-07-20
- 이전 상태: 2.6 예시 주석(`:374`)이 "`wafer_count`/`defect_count`/`normal_count`는 **전체 25장 기준**"이라 서술해 로트당 웨이퍼가 항상 25장인 것처럼 읽혔고, `wafer_id` 형식 서술(`:378`)도 "`"1"`~`"25"`, 로트당 최대 25장"으로 상한과 고정값의 구분이 흐렸다
- 변경 후 상태: 로트당 웨이퍼 수는 **가변(1~25장)**이며 25장은 상한이라는 점을 두 곳(`:374`·`:378`) 모두에 명시. 집계 3필드는 "25장 기준"이 아니라 **"해당 로트 전량 기준"**으로 재서술하고, 예시의 `wafer_count:25`는 25장짜리 로트를 가정한 값임을 괄호로 밝힘. 프론트 지침으로 "그리드 슬롯 25칸 고정 렌더 금지, `wafers.length`에 맞춰 렌더"를 추가
