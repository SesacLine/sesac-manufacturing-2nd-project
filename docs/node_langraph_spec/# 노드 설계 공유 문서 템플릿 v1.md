## 1. 입출력 계약 (필수)

> 이 절이 이 문서의 핵심이다. **다른 노드 담당자가 실제로 읽는 유일한 부분**이라고 생각하고 써라.
> 타입은 `backend/state.py`의 TypedDict 이름을 그대로 인용한다. 새로 정의했으면 여기에 적고 state.py에도 올린다.

### 1-1. 입력

| state 키 | 타입 | 채우는 주체 | 없으면/None이면 |
|---|---|---|---|
| `pattern` | `str` | ② grouper | (필수 — 없으면 동작 불가 / 폴백 있음) |
| `observation` | `Observation \| None` | ③ vlm_describe | (예: None이어도 동작, enum 진입만 사용) |

- **입력에 대해 내가 가정하는 것(전제조건)**: 예 — "`pattern`은 반드시 KG DefectPattern 3종 중 하나거나 `Unknown`이다. `Normal`은 애초에 안 들어온다(② 계약)."
- **가정이 깨지면**: 예외를 던지는가 / 빈 결과를 내는가 / 조용히 통과하는가?

### 1-2. 출력

| state 키 | 타입 | 비고 |
|---|---|---|
| `candidates` | `list[GraphRAGCandidate]` | 빈 리스트 가능(→ 조건부 엣지가 `unmapped`로 분기) |

- **하위 노드가 믿어도 되는 불변식**: 예 — "`candidates`는 `rank` 오름차순 정렬이 보장된다",
  "`tier='자동'`인 원소는 `evidence`가 절대 None이 아니다".
  → **여기에 쓴 문장이 곧 테스트 케이스**가 된다. 못 지킬 약속은 쓰지 말 것.
- **내가 건드리지 않는 state 키**: (누가 봐도 명확하면 생략 가능)

### 1-3. 새로 도입/변경한 필드

기존 `state.py`에 없던 필드를 추가했다면 필드명 · 타입 · **왜 필요했는지** 한 줄씩.
(없으면 "없음")

## 4. 설계에서 중요하게 고려한 것 (필수)

> **이 문서를 만드는 진짜 이유가 이 절이다.** "무엇을 했나"가 아니라 "왜 그렇게 했나".
> 항목마다 아래 3줄 구조로:

### 4-1. (고려사항 제목 — 예: "KG 순위를 최종 순위로 쓰지 않은 이유")

- **문제**: 무엇이 곤란했나 (가능하면 실측 수치·실패 사례와 함께)
- **선택**: 어떻게 했나
- **대안과 기각 이유**: 검토했다가 버린 방법 (없으면 "대안 미검토"라고 솔직히)
- **되돌릴 조건**: 어떤 상황이 되면 이 결정을 다시 봐야 하나

(2~4개 정도. `docs/BACKEND_DECISIONS.md`에 이미 올린 결정은 **번호만 인용**하고 여기서 반복하지 말 것.)

## 5. 외부 의존 (LLM · MCP · 파일 · DB)

| 무엇 | 어디 | 결정적인가 | 없으면 |
|---|---|---|---|
| 예: `hypotheses.json` | `kg_rca/outputs/` | 결정적 | 노드 동작 불가 |
| 예: LLM 호출 | `create_react_agent` | **비결정적** | 폴백 경로 있음/없음 |
| 예: MCP `query_telemetry` | 싱글턴 세션 | 결정적 | 증거 미수집 |

- **비결정적 요소가 있으면 반드시 명시**한다 — 테스트 전략과 재현성이 여기서 갈린다.
- 호출 횟수 / 대략적 지연 (예: "그룹당 1콜, Center 244건 기준 ~40초").
