# LangGraph 골격 리팩터 설계 공유 문서 v1.0

> **목적**: 배치 파이프라인의 LangGraph 골격이 이번에 왜·어떻게 바뀌었는지, 그리고
> 다른 파트가 이 골격 위에서 작업할 때 **무엇을 믿어도 되는지(계약)**를 코드를 열어보지
> 않고도 알 수 있게 한 문서다. 코드 설명서가 아니라 **구조 변경의 계약서 + 설계 의도**다.
> **작성**: 2026-07-24. 대상 작업 = 골격 서브그래프화(#32, PR #38 머지 완료) + 시그니처
> 평탄화(#33). **성격은 두 번 다 "행위 보존 리팩터"** — 결과물은 이전과 같아야 하고,
> 실제로 같음을 골든 대조로 실증했다(§5).
> 내부 정책 결정은 `docs/BACKEND_DECISIONS.md`, 이전에 재작성된 ⑤·⑥ 노드 설계는
> `docs/hypo_critic_설계공유_v1.0.md` 참고.

## 0. 요약

배치 파이프라인 ⓪~⑦ 중 **그룹 단위로 도는 뒷부분(④~⑦)의 실행 구조**를 두 단계에 걸쳐
정리했다. 결과물(분석 카드·판독 결과·커서)은 **한 글자도 바뀌지 않았고**, 바뀐 것은 그
결과에 이르는 골격뿐이다.

- **#32 — 2층 구조로 재편**: 그룹별 반복을 LangGraph "바깥 배치 그래프 + 안쪽 그룹 서브그래프"
  두 층으로 나눴다. 예전엔 그래프 밖 파이썬 반복문에 숨어 있던 그룹 팬아웃과, 응답 노드 안
  `if` 문에 숨어 있던 "재료 없으면 LLM 안 부른다" 판단을, **그래프 위상 자체(조건부 엣지)로
  끌어올렸다.** 이게 환각 억제를 코드 구조로 보장하는 핵심 장치다.
- **#33 — 임시 다리(어댑터) 제거**: #32 때는 노드 함수의 옛 모양을 그대로 두고 "어댑터"라는
  임시 연결층으로 새 구조에 이어 붙였다. 그 임시 다리를 걷어내, 노드가 자기 그룹 하나의
  상태를 **직접** 받도록 정리했다. 곁가지로 진행표시가 한 칸 건너뛰던 누락 1건도 메웠다.

두 단계 모두 **함수 내부 알고리즘(증거 수집·랭킹·판정 규칙·정렬)은 한 줄도 건드리지 않았다.**
바뀐 것은 "누가 무엇을 받아서 어디로 넘기느냐"라는 배선뿐이다.

## 1. 무엇이 바뀌었나 — Before / After

### 1-1. 파이프라인에서의 위치

배치 실행은 ⓪~⑦ 순서다. **⓪~③은 배치 전체를 한 번씩** 돌고, **④~⑦은 결함 패턴 그룹마다
한 번씩** 돈다. 이번 리팩터는 이 "그룹마다 도는 ④~⑦" 부분의 구조를 다뤘다.

```
[배치 그래프 — 배치당 1회]
⓪ 저수율 로트 선별 → ① 웨이퍼맵 판독(CNN) → ② 패턴별 그룹화 → ③ 그룹 관측 → run_groups
                                                                              │
                                          그룹마다 아래 서브그래프를 순차 호출 │
                                                                              ▼
[그룹 서브그래프 — 그룹당 1회]
④ KG 원인후보 조회
   ├─(후보 0건)───────────────→ ⑦' LLM 없이 "판단불가" 응답      ← 조건부 엣지 ①
⑤ 증거 수집·검증·재랭킹
⑥ 규칙 게이트(채택/기각/보류)
   ├─(채택 0건)───────────────→ ⑦' LLM 없이 "근거부족" 응답      ← 조건부 엣지 ②
⑦ 응답 카드 생성
```

### 1-2. Before — 리팩터 이전에 곤란했던 것

- **그룹 팬아웃이 그래프 밖에 있었다.** 그룹별 반복이 LangGraph 그래프가 아니라 별도 파이썬
  반복문에 있어, "이 파이프라인이 그룹마다 뭘 하는가"가 그래프 그림에 드러나지 않았다.
- **조건 분기가 노드 안에 숨어 있었다.** "KG 후보가 0건이거나 채택된 가설이 0건이면 LLM을
  부르지 않고 판단불가로 답한다"는 이 프로젝트의 환각 억제 규칙이, 응답 노드 내부 `if` 두
  개에 묻혀 있었다. 즉 **그래프만 봐서는 그 안전장치가 안 보였다.**
- **그룹별 결과를 합칠 때 조용히 유실될 위험이 있었다.** 그룹 결과를 담는 필드에 병합 규칙이
  없어, 나중에 그룹을 병렬로 돌리면 뒤 그룹이 앞 그룹 결과를 덮어쓸 수 있는 구조였다.

### 1-3. After — #32가 바꾼 것 (2층 구조)

- **2층으로 분리**: 배치 전체를 도는 바깥 그래프와, 그룹 하나를 도는 안쪽 서브그래프로 나눴다.
  바깥 그래프는 그룹들을 **순차로** 서브그래프에 태운다(병렬화는 뒤 §6 참고).
- **조건 분기를 그래프 위상으로 승격**: 위 그림의 "조건부 엣지 ①·②"가 그것이다. 재료가
  없으면(후보 0건·채택 0건) **LLM 응답 노드로 가는 길 자체가 그래프에 존재하지 않는다.**
  나중에 ⑦에 실제 LLM이 붙어도, 재료 없이 문장을 지어내는 경로는 위상적으로 차단된다.
- **그룹 결과 필드에 병합 규칙 부착**: 그룹별 결과를 합칠 때 키 기준으로 병합하도록 해,
  병렬화하더라도 서로 덮어쓰지 않게 했다.
- **진행표시·로그 정비**: 2층 스트림 신호를 올바로 풀어 진행률이 단조 증가하게 하고, 로그에
  `[Center]`처럼 어느 그룹 처리 중인지 태그가 붙게 했다(순차·병렬 양쪽에서 안전).
- **단, 이때는 노드 함수의 옛 모양을 유지**했다. 서브그래프와 옛 노드 사이를 "어댑터"라는
  임시 연결층이 이어줬다 — 그룹 하나짜리 **가짜 배치 상태**를 만들어 옛 노드에 넘기는 방식.

### 1-4. After — #33이 바꾼 것 (임시 다리 제거)

- **어댑터(임시 다리)를 걷어냈다.** 노드가 이제 자기 그룹 하나의 상태를 **직접** 받는다.
  가짜 상태를 재구성하던 코드, 결과를 `{그룹ID: 값}`으로 한 겹 감싸던 중첩, 그리고 더는
  필요 없는 그룹ID 인자가 동시에 사라졌다.
- **도달 불가가 된 방어 코드 2개도 함께 삭제**했다. 예전엔 상태가 비어 있을 때 조용히
  "unknown" 카드를 만들던 폴백이 있었는데, 새 구조에선 필수 값이 항상 채워진 채로 들어오므로
  이 폴백은 실행될 수 없다. 없앤 결과, 만에 하나 값이 빠지면 **조용히 잘못된 카드를 만드는
  대신 즉시 오류로 드러난다**(리뷰에서 "개선"으로 확인).
- **진행표시 누락 1건 복원**: 다른 파트(#40)가 CNN·관측 배선을 마치면서 진행표시 표에 ③
  관측 단계를 빠뜨려, ③이 끝나면 진행률이 2에서 4로 건너뛰었다. 이 칸을 메웠다.

## 2. 왜 이렇게 했나 — 설계 의도

### 2-1. 조건 분기를 "노드 안 if"가 아니라 "그래프 조건부 엣지"로 올린 이유

- **문제**: 환각 억제 규칙("재료 없으면 LLM 안 부른다")이 노드 내부에 숨으면, 그래프만 봐선
  안전장치의 존재를 알 수 없고, 나중에 누가 노드를 고치다가 그 `if`를 무심코 흐트러뜨릴 수 있다.
- **선택**: 그 판단을 그래프의 조건부 엣지로 끌어올려, **"LLM 응답으로 가는 경로 자체를
  없애는"** 방식으로 바꿨다. 재료가 없으면 애초에 그 길이 없다.
- **효과**: ⑦에 실제 LLM을 붙이는 건 이후 과제인데, 그때도 안전장치가 위상으로 박혀 있어
  "재료 없이 문장을 쓰는" 실수 경로가 원천 차단된다.

### 2-2. "행위 보존 리팩터"로 못박은 이유 — 결과가 바뀌면 실패다

- **문제**: 구조를 바꾸는 김에 기능도 손대면, 나중에 결과가 달라졌을 때 그게 구조 변경 탓인지
  기능 변경 탓인지 가릴 수 없다.
- **선택**: 두 단계 모두 **결과물(분석 카드·판독·커서)이 이전과 동일할 것**을 성공 기준으로
  삼고, 실제 배치를 리팩터 전/후로 돌려 산출물을 대조했다(§5). 실모델 활성화·기능 추가는
  전부 범위 밖으로 밀어냈다.
- **되돌릴 조건**: 결과가 달라지면 그건 개선이 아니라 회귀다 — 즉시 원인을 찾는다.

### 2-3. 어댑터(임시 다리)를 먼저 뒀다가 나중에 걷어낸 이유

- **문제**: #32에서 서브그래프를 떼어낼 당시, ④~⑦ 노드 내부는 다른 이슈(#22, Hypothesis·Critic
  재작성)에서 대대적으로 다시 쓰이는 중이었다. 골격 변경과 노드 재작성을 동시에 하면 서로
  충돌하고 리뷰가 어려워진다.
- **선택**: 두 작업을 분리하기 위해, 골격만 먼저 새로 짜고 옛 노드는 **어댑터라는 임시 다리**로
  이어 붙였다. #22가 머지돼 블로커가 풀린 뒤(#33), 그제서야 다리를 걷어내 노드 시그니처를
  새 구조에 맞게 평탄화했다.
- **왜 반드시 걷어내야 했나**: 어댑터를 남겨두면 가짜 상태 재구성·불필요한 중첩·쓰지 않는
  인자가 계속 코드에 남아, **다음 사람이 진짜 계약을 오독하기 쉽다.**

### 2-4. 의존성 주입 방식과 후속 주의 (기술적 함정 — 반드시 인지)

- 노드가 필요로 하는 외부 도구(KG 클라이언트·MCP)는 그래프를 조립하는 시점에 미리 묶어
  넣는다. 리뷰에서 라이브러리 동작을 직접 확인해 **현재 실행 경로가 정상**임을 검증했다.
- **⚠️ 후속 주의(중요)**: 이 주입 방식 때문에, 나중에 노드가 "다음에 어디로 갈지"를 스스로
  반환값으로 지정하는 방식(`Command` 라우팅)으로 바뀌면, 그래프가 분기 대상을 자동으로
  추론하지 못한다. **그때는 조립 시점에 분기 대상 목록을 명시적으로 넘겨야 한다.** 지금
  구조에선 문제없지만, 라우팅 방식을 바꾸는 사람은 이 점을 반드시 기억할 것.

(위 결정들의 배경·기각 대안은 `docs/BACKEND_DECISIONS.md`, 조립·라우팅의 실제 흐름은
루트 `CLAUDE.md` "파이프라인" 절 참고.)

## 3. 실패·경계 케이스 계약

| 상황 | 지금 동작 | 다른 파트가 보게 되는 것 |
|---|---|---|
| KG 후보가 0건인 그룹 | ⑤⑥을 건너뛰고 조건부 엣지로 곧장 "판단불가" 응답 | LLM 미호출, "unmapped(매핑 불가)" 성격의 카드 |
| 채택된 가설이 0건인 그룹 | ⑦(LLM) 대신 "근거부족" 응답 | LLM 미호출, "insufficient(근거부족)" 성격의 카드 |
| `Normal`(정상) 패턴 그룹 | 애초에 서브그래프에 태우지 않음 | 그룹 자체가 생기지 않음(정상 웨이퍼) |
| 그룹 상태에 필수 값이 빠진 경우 | (리팩터 후) 조용히 넘어가지 않고 **즉시 오류로 중단** | 잘못된 "unknown" 카드가 만들어지지 않음(개선) |

- **재시도는 없다.** 채택 후보가 0개면 재시도하지 않고 곧바로 "판단불가"로 답한다(기획안 방침).
  이 가드가 이제 노드 안 `if`가 아니라 **그래프 위상**에 박혀 있다는 게 이번 변경의 핵심이다.
- 그룹은 **순차** 실행이라 그룹끼리 서로 간섭하지 않는다.

## 4. 다른 파트가 믿어도 되는 계약

이 골격 위에서 작업하는 사람이 알아야 할 불변식이다. **여기 적힌 문장이 곧 테스트로 박혀 있다.**

- **그룹 하나의 상태(`GroupState`)가 서브그래프의 단위다.** ④~⑦ 노드는 배치 전체가 아니라
  이 그룹 상태 하나만 받고, 자기 그룹의 결과 필드만 돌려준다.
- **서브그래프의 모양(위상)은 불변이다.** 노드 5종의 이름·진입점·조건부 엣지 2종의 분기
  대상이 그대로다. 리팩터가 바꾼 건 노드가 상태를 받는 방식뿐, 흐름이 아니다.
- **바깥 그래프의 배선(⓪~③)은 이번에 건드리지 않았다.** CNN·관측 담당(#40) 영역과는
  경계가 분리돼 있다.
- **그룹별 결과를 배치로 합치는 방식(누적 계약)도 불변이다.** 그룹 결과를 그룹ID 기준으로
  병합하고, `Normal` 그룹을 거르고, 로그에 그룹 태그를 붙이는 동작이 그대로다.
- **함수 이름·어휘는 유지**했다(예: `graphrag`라는 옛 이름). 실제로는 KG 조회지만 이름 변경은
  별도 단계(step 8)로 미뤘으니, 코드를 읽을 때 이름과 실제 역할의 간극에 유의할 것.

## 5. 검증 방법 — "결과가 같다"를 어떻게 증명했나

리팩터의 성공 기준은 "결과물이 이전과 동일"이므로, 두 겹으로 확인했다.

- **1차(구조 확인)**: 시그니처·반환 형태·노드 등록·진행표시 매핑을 테스트로 전량 확인.
  최종 **125건 통과, 실패 0**(fab.db 없이 CI에서 도는 것과 동일 조건).
- **2차(행위 보존 실증)**: 실제 fab 데이터로 배치를 리팩터 전/후로 돌려 산출물을 대조.
  - #32에서는 산출물이 **완전히 바이트 단위로 일치**했다.
  - #33에서는 한 가지 주의가 있다. #22 머지로 ⑤에 LLM 에이전트가 들어오면서, 최종 카드의
    **서술 문장(rationale)** 한 필드만은 같은 입력에도 매번 미세하게 달라진다(LLM 특성,
    temperature=0이어도 재현 안 됨). 그래서 **같은 코드로 2회 캡처해 "차이가 그 서술 필드
    하나로만 국한됨"을 먼저 실측**한 뒤, 그 필드를 제외한 나머지(증거 숫자·판정·정렬·요약)의
    지문(SHA256)이 **일치함**을 성공 기준으로 정의했다. 실제로 일치했다.
  - 규모 지표도 함께 일치: 대상 로트 67 · 그룹 1(Center) · 판독 1377 · 가설 375건
    (채택 25 / 기각 21 / 보류 329).
- **비결정 요소는 rationale 문장 하나뿐**이라는 것이 이 검증의 핵심 결론이다. 판정에 쓰이는
  숫자·규칙 결과는 전부 결정적이다.

## 6. 알려진 한계 · 후속 (의도적으로 미룬 것들)

이번 두 단계는 **순수 구조 변경**이라, 아래는 성격이 다르다는 이유로 명시적으로 다음 단계로
미뤘다. "안 한 것"이 아니라 "일부러 분리한 것"이다.

| 항목 | 지금 상태 | 이유 / 담당 |
|---|---|---|
| 이름 리네임 `graphrag → kg` | 옛 이름 유지 | 순수 리네임은 별도 단계(step 8), 보류 확정 |
| 노드 실패 계약(try/except) | 미도입 | 예외처리 추가는 "순수 구조 변경" 성격을 깨서 diff에서 진짜 변경을 묻음 → 별도 단계 |
| 타입 이름 `VLMResult → CNNResult` | 옛 이름 유지 | 범위 밖(별도 정리) |
| 관측 자연어(`description`) 실값 배선 | 빈 값 | 비전 담당(#40 후속). VLM 자연어 실연동과 함께 |
| 그룹 **병렬** 실행(Send) | 순차 유지 | MCP 싱글턴 세션 제약. MVP 검증 후 재검토 |
| ⑦ 응답생성 LLM 실연동 | 템플릿 유지 | 조건부 엣지가 이미 재료 없는 호출을 막아둠 |
| `Command` 라우팅 전환 시 분기 대상 명시 | 해당 시 필요 | §2-4의 후속 함정 — 라우팅 방식 바꾸는 사람이 인지할 것 |
