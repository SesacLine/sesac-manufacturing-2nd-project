`evidence` 값만 읽어서 판정한다(코드·다른 문서에서 이 성질을 "faithfulness firewall"이라 부른다).
⑤가 모든 수치를 도구 반환값에서 계산해 두었기 때문에(옵션 A), ⑥은 그 값만 봐도 사실에 근거한
판정이 된다. LLM이 잘못된 문장을 써도 판정에는 들어오지 않는다.

**이 문서에서 쓰는 용어** (자세한 정의는 부록 B):
`tier`(검증등급) · `investigated`(조사 여부) · `judge_unknown`(판단 보류) · `함정`(trap) ·
`reject_token`(고정 사유 코드).

## 1. 입출력 계약 (필수)

이 노드는 그룹 하나마다 1회 실행된다. 아래 키는 그룹 단위 상태(`GroupState`) 기준이다.

### 1-1. 입력

| state 키 | 타입 | 채우는 주체 | 없으면 / None이면 |
|---|---|---|---|
| `hypotheses` | `list[Hypothesis]` | ⑤ build_hypotheses | 빈 리스트면 채택 0건 → status `insufficient_evidence` |
| `mcp` (주입, **미사용**) | `MCPClient` | `deps.py` | 받기만 하고 호출하지 않는다(재조회 안 함) |

- **전제조건**: 각 가설의 `evidence`는 ⑤가 옵션 A로 채워 둔 상태다(수치는 전부 도구 출처).
  `investigated`(조사 여부) 마커가 실려 있다.
- **가정이 깨지면**: `evidence`에 특정 키가 없어도 `.get()`으로 `None` 처리한다(관대한 읽기).
  대부분의 규칙은 값이 `None`이면 "통과" 또는 "보류"로 안전하게 흐르므로 예외를 던지지 않는다.

### 1-2. 출력

| state 키 | 타입 | 비고 |
|---|---|---|
| `critic_result` | `CriticResult` = `{status, accepted, rejected}` | `status`는 `accepted` 또는 `insufficient_evidence` |

**하위 노드(⑦)가 믿어도 되는 불변식**:
- **fab 재조회 0회.** ⑥ 구간에서 MCP·LLM 호출이 전혀 없다 — 실측으로도 0회.
- **`accepted`의 순서 = 입력(⑤) 순서를 그대로 보존한다.** ⑤가 1위로 올린 가설이 `accepted[0]`,
  곧 대표원인이 된다(순위를 바꾸지 않는다).
- 기각·보류된 행마다 `reject_token`(고정 코드)과 `reject_reason`(표시용 문구, #45로 영문)이 붙는다.
- `status`는 채택이 1건 이상이면 `accepted`, 0건이면 `insufficient_evidence`다. **재시도 없음.**
- **내가 건드리지 않는 키**: `hypotheses`(순서·내용 그대로 통과), `final_response`.

### 1-3. 새로 도입/변경한 필드

| 필드 | 타입 | 왜 필요한가 |
|---|---|---|
| `critic_result` | `CriticResult` | ⑦ 응답 카드가 읽는 판정 결과(채택/기각/보류 3분류 + status) |
| `reject_token` | `str` (enum) | ⑦이 프론트 표시값(verdict)을 매핑하는 **유일한 키**. 자연어 매칭 금지 |
| `reject_reason` | `str` | 사람이 읽는 기각 사유 문구(표시용). 판정 로직에는 안 쓴다 |

## 2. 실패·경계 케이스 계약 (필수)

| 상황 | 이 노드의 동작 | 하위 노드(⑦)가 보게 되는 것 |
|---|---|---|
| 입력 `hypotheses`가 빔 | `accepted=[]`, `rejected=[]`, status `insufficient_evidence` | ⑦'가 "판단 불가" 카드 |
| `evidence` 필드 결측 | `.get()`으로 `None` 처리 → 규칙별 안전 분기(반대근거 None이면 기각, drift None이면 미조사 시 보류) | 정상 판정 흐름 |
| 외부 호출 실패 | **해당 없음** — 이 노드는 MCP/LLM/파일을 호출하지 않는다 | 항상 결정적 |
| 채택 0건 | status `insufficient_evidence`, 재시도 안 함 | ⑦'가 "판단 불가" 카드 |

- **예외를 던지는 경우**: 없다. 순수 함수라 관대한 읽기로 예외 없이 판정까지 간다.
- **타임아웃·재시도**: 없음 — 외부 의존이 0이라 지연·재시도 개념 자체가 없다.

## 3. 내부 플로우

가설 1건마다 아래 규칙을 **위에서부터** 검사한다. **처음 걸리는 규칙이 판정을 정하고**, 끝까지
아무것도 안 걸리면 채택한다. 전부 결정론이며 fab 재조회는 없다.

```
가설 1건
 ① 시간정합     정비 시각 > 결함 시각 (정비가 결함보다 늦음)?     → P2 rejected   ← 함정 기각 지점
 ② 반대근거     normal_ratio를 수집 안 함(None)?                → P3 rejected
 ③ faithfulness 자동 tier + 조사됨인데 drift 판정이 비어 있음?     → P4 rejected
 ④ KG 메커니즘   tier == 근거없음?                              → P5 judge_unknown (보류)
 ⑤ 미조사       investigated == False?                        → judge_unknown (보류)
 else                                                          → accepted

status = accepted 1건 이상 → "accepted",  0건 → "insufficient_evidence"
```

판정은 **3분류**로 나뉜다:
- **accepted** — 증거가 지지함.
- **rejected** — 증거가 반박함 (P2·P3·P4).
- **judge_unknown** — 조사 못 했거나 확인 불가라 판단 보류 (P5·미조사). **기각과 다르다.**

- **⑤(미조사)가 체인 맨 뒤인 것이 핵심 설계다** → §4-2.

## 4. 설계에서 중요하게 고려한 것 (필수)

### 4-1. 판정에 LLM을 쓰지 않는다 — ⑤ evidence만 읽는다

- **문제**: 판정 단계에 LLM이 개입하면, `evidence`에 없는 값으로 채택·기각할 여지가 생긴다(환각).
  실행마다 결과가 달라져 평가 재현성도 깨진다.
- **선택**: 규칙 기반 결정론으로 하고, **fab을 다시 조회하지 않고 ⑤가 채운 `evidence`만** 읽는다.
  ⑤가 모든 수치를 도구 반환에서 계산해 두었기(옵션 A) 때문에 이 방식으로도 사실 근거 판정이 된다.
- **대안과 기각**: LLM으로 판정(LLM-judge) → 재현 불가, "evidence만 읽는다"는 성질이 무너짐.
- **되돌릴 조건**: ⑦에 LLM이 붙어 "문장 단위 사실 대조"가 필요해지면 **별도 층**으로 설계한다 —
  지금의 규칙 게이트(재조회 없음)는 그대로 둔다(§8).

### 4-2. 반박이 보류보다 먼저 판정된다 — 규칙 순서의 의미

- **문제**: `반자동` 후보는 지금 항상 "미조사" 상태다. 만약 미조사 검사를 체인 앞에 두면,
  시간역전(P2) 규칙이 적용되기도 전에 함정 후보가 전부 "보류"로 빠져 명시 기각을 못 한다.
- **선택**: 미조사 검사(⑤)를 체인 **맨 뒤**에 둔다. 정비 시각(`maintenance_ts`)은 ⑤의 결정론
  단계(pre-pass)가 수집한 사실이라, 미조사 여부와 무관하게 시간정합(①) 판정에 유효하다. 그래서
  함정 장비의 시간역전은 보류가 아니라 명시 기각으로 먼저 걸린다.
- **되돌릴 조건**: 규칙을 추가할 때 "반박은 앞, 보류는 뒤"라는 순서 불변식을 유지해야 한다.

### 4-3. "반박됨"과 "조사 안 됨"을 구분한다 (judge_unknown)

- **문제**: 조사하지 못한 후보를 반박된 후보와 똑같이 기각하면, 억울하게 죽는 후보가 생기고
  "채택 0건"이라는 판단 불가 상태가 부정확해진다.
- **선택**: 판정을 3분류로 나눠, 확인 못 한 것은 `judge_unknown`(보류)로 따로 뺀다. 보류는
  기각이 아니다 — "아직 모른다"는 정직한 상태다.
- **되돌릴 조건**: `반자동` 실제 조사 경로가 생기면 그 보류 토큰(SEMI_AUTO_PENDING)을 걷어낸다.

### 4-4. 표시 문구가 아니라 고정 코드로 판정을 전달한다 (API §2.7)

- **문제**: ⑦이 프론트 표시값(verdict 3종)을 자연어 사유 문구로 매칭하면, 문구를 조금만 고쳐도
  프론트 표시가 깨진다.
- **선택**: 기각·보류 행에 **고정 코드(`reject_token`)**를 부여하고, ⑦은 그 코드만 보고 매핑한다.
  `reject_reason` 문구는 순수 표시용이라 언제든 바꿔도 안전하다 — 그래서 #45의 영문화가 프론트를
  깨지 않았다.
- **대안과 기각**: 사유 문자열 직접 매칭 → 다국어·문구 수정에 취약.

## 부록 B. 용어

| 용어 | 뜻 (Plain) |
|---|---|
| tier(검증등급) | KG가 후보마다 매긴 "어떤 도구로 확인 가능한가" 등급. `자동`(센서값)/`반자동`(정비·레시피)/`근거없음` |
| investigated | ⑤가 그 후보를 실제로 도구 조회했는지 여부(bool). False면 ⑥이 판단 보류 |
| judge_unknown(판단 보류) | 조사 못 했거나 확인 불가라 채택도 기각도 하지 않는 상태. **기각과 구분한다** |
| faithfulness firewall | ⑥이 fab을 다시 조회하지 않고 ⑤가 채운 `evidence`만 읽는다는 계약. 잘못된 LLM 서술이 판정에 못 들어오게 한다 |
| 함정(trap) | 상관관계는 높지만 실제 원인이 아닌 장비. 예: 불량 로트 전부가 지났지만 정비 시각이 결함보다 늦은 장비 — 상관은 있어도 시간상 원인일 수 없다 |
| reject_token | 기각·보류 사유를 나타내는 고정 코드. ⑦이 프론트 표시값을 정하는 유일한 키(문구가 아니라 코드로 매핑) |
| P2 / P3 / P4 / P5 | 규칙 체인의 사유 코드 번호. P2=시간역전, P3=반대근거 미수집, P4=faithfulness, P5=KG 근거없음 |
