# 데이터 모델 설계_v2.0

> 갱신: 2026-07-13 · v1.0(07-08 초안, 안지운)을 **KG 실제 구현(schema v2.4)** 에 맞춰 개정
> KG 상세 정본: `../docs/KG_schema_v1.2.md` / 출력 계약: `KG_output_명세.md` / 정합성: `MCP_KG_정합성검토.md`
> v1.0과 달라진 부분은 각 절에 **[v1.0 대비]** 로 표시. §6이 앞뒤 컴포넌트 수정 체크리스트.

데이터 저장소는 3종류다 — ① fab.db(SQLite) ② Neo4j(그래프 DB) ③ app_state.db(SQLite) + LangGraph `RCAState`(실행 중 메모리 상태). 여기에 **KG 출력 파일(`outputs/hypotheses.json`)이 Neo4j와 LangGraph 사이의 인터페이스**로 추가된다.

```
fab.db ──(시뮬레이터가 빌드)──> MCP 서버(T1~T9) ──> LangGraph 서비스
                                                        ▲
문헌(data/raw/) ──(KG 파이프라인 0~5번)──> Neo4j ──(6번)──> hypotheses.json
                                                (③ 노드가 읽는 계약)
```

**[v1.0 대비] 주의**: KG 저장소의 `fab.md`는 `lot_history`/`telemetry`/`alarm`/`maintenance` 4테이블만 기록하고 있어 `metric_series`·`wafer` 테이블과 어긋난다 — §6 체크리스트 F-2.

## 2. Neo4j 그래프 스키마 — **[v1.0 대비 전면 교체]**

v1.0의 "매핑 지식을 MERGE 시딩" 설계는 **문헌 추출 파이프라인**으로 대체됐다.
그래프는 문헌(교과서 표·논문)에서 LLM이 추출한 지식이고, `mapping_table.yaml`(패턴별 큐레이션
원인 + prob)은 그래프에 넣지 않고 **출력 조립 시점의 오버레이**로만 얹는다 — 문헌 근거와
큐레이션 근거의 provenance를 섞지 않기 위함이다.

### 노드 라벨 (실제)

| 라벨 | 유일키 | 주요 속성 | 출처 |
| --- | --- | --- | --- |
| `DefectPattern` | `id` (`Center`/`Edge-Ring`/`Scratch`) | `aliases` | **시드** (VLM 클래스와 문자열 일치) |
| `ProcessStep` | `id` (`ETCH` 등 6종) | `aliases` | **시드** (`lot_history.step` join key) |
| `Parameter` | `id` (`rf_power` 등 20종) | `steps`, `aliases`, `fab_table` | **시드** (`telemetry.param` join key) |
| `SpatialSignature` | `id` = `{shape}@{zone}` | `shape`, `zone` (코드 enum으로 닫힘) | 문헌 추출 |
| `FailureMode` | `id` | `name`, `description` | 문헌 추출 |
| `Cause` | `id` | `name`, `description`, `unverifiable_signals` | 문헌 추출 |
| `Maintenance` | `id` | `name`, `consumable`, `fab_table` | 문헌 추출 |
| `Recipe` | `id` | `name`, `fab_table` | 문헌 추출 |

`Parameter`/`Maintenance`/`Recipe`는 공통 슈퍼라벨 **`:Evidence`** 를 갖는다 (검증 종착점).
v1.0의 `TelemetrySignature` 노드는 `Parameter` + `VERIFIED_BY.direction`으로,
`Citation` 노드는 **관계 속성**(`quotes`, `chunk_ids`)과 오버레이의 `mapping.citation`으로 대체됐다.

### 관계 타입 (실제)

| 관계 | 방향 | 속성 | v1.0 대응 |
| --- | --- | --- | --- |
| `ARISES_IN` | `(DefectPattern)→(ProcessStep)` | `occurrence_prior`, `quotes`, `chunk_ids` | (없음 — 신규 층) |
| `HAS_SIGNATURE` | `(DefectPattern)→(SpatialSignature)` | `quotes`, `chunk_ids` | (없음) |
| `FORMS_IN` | `(SpatialSignature)→(ProcessStep)` | `occurrence_prior`, ... | (없음) |
| `OCCURS_IN` | `(FailureMode)→(ProcessStep)` | ... | `IN_PROCESS` 유사 (주체가 다름) |
| `CAUSED_BY` | `(FailureMode)→(Cause)` | ... | ⚠ v1.0은 `(Pattern)→(Cause)`였음 — **방향·주체 변경** |
| `ATTRIBUTED_TO` | `(DefectPattern)→(Cause)` | ... | v1.0 `CAUSED_BY`에 가장 가까움 (단 `prob` 없음) |
| `VERIFIED_BY` | `(Cause)→(:Evidence)` | `target_label`, `direction` | `HAS_SIGNATURE` 대체 |

⚠ **`prob`는 그래프에 없다.** v1.0의 `CAUSED_BY {prob}`에 있던 배정 확률은 `mapping_table.yaml`이
정본이고, KG 출력의 `mapping.prob`로만 노출된다 (오버레이 매칭이 된 가설에 한함 → null 가능).

### 시딩과 갱신 — **[v1.0 대비 변경]**

- v1.0: "1회성 MERGE 시딩 스크립트" → 실제: **파이프라인** `0_reset → 2_load → 3_split → 4_ingest(시드 3앵커) → 5_build(LLM 추출 + 검증 규칙 + 앵커 보강 패스)`
- 문헌이 추가되면 재적재가 일어난다. **재적재 간 진입점 엣지가 변동할 수 있으므로**(추출 비결정성),
  서비스는 그래프를 직접 믿지 말고 스냅샷(`hypotheses.json`)의 `meta.generated_at`으로 신선도를 검사한다.

### 런타임 조회 — **[v1.0 대비 변경: 고정 Cypher → 출력 파일 계약]**

v1.0의 "코드에 박아둔 파라미터 Cypher"는 실제 스키마에서 동작하지 않는다(노드·관계가 다름).
대신 **6번(`6_ask_graphrag.py`)이 결정적 순회로 `outputs/hypotheses.json`을 생성**하고,
서비스(③ 노드)는 이 파일에서 `questions[].pattern` 섹션을 읽는다. LLM Cypher 생성 없음은 동일.

가설 1건의 핵심 키 (전체는 `KG_output_명세.md`):

```
tier            자동|반자동|근거없음   ← 검증 방법 분류 (agent 판정 가능 여부)
scenario_hint   A3|A5|A2|A6|null      ← MCP 검증 체인 라우팅
path            {signature, step, failure_mode, cause, evidence, evidence_label}
verification    {fab_table, direction, unverifiable_signals}
score           {occurrence_prior, evidence_docs, evidence_chunks}  ← 전부 측정값
provenance      {chunk_ids, quotes}
mapping         {matched_cause, prob, citation, ...} | null   ← 큐레이션 오버레이 출처
```

순위 = `(occurrence_prior, evidence_docs, evidence_chunks)` 내림차순 — **prob 정렬이 아니다**.
v1.0의 `ORDER BY r.prob DESC`에 해당하는 것이 필요하면 `mapping.prob`가 있는 가설만 골라 쓴다.

### 다중 홉 질의 *(v1.0 유지, 스키마만 갱신)*

```cypher
// "CMP가 여러 패턴의 원인 공정에 공통으로 등장하는가"
MATCH (p:DefectPattern)-[:ARISES_IN]->(s:ProcessStep {id:"CMP"})
RETURN p.id
```

## 3. LangGraph `RCAState` — **[v1.0 골격 유지, ③④ 계약 변경]**

필드 표는 v1.0과 동일하되, KG와 닿는 부분이 바뀐다:

| 필드 | v1.0 | **v2.0 (변경)** |
| --- | --- | --- |
| `graphrag_candidates[group_id]` | ③이 고정 Cypher로 조회한 `{pattern, candidates:[...]}` | ③이 **`hypotheses.json`의 해당 pattern 섹션**에서 구성. 후보 요약 수준(시나리오는 "후보 3종" 기대)은 Q3 `candidates[]` 뷰 확정 전까지 **`mapping`이 붙은 가설 + tier별 상위 N**으로 구성 권장 |
| `hypotheses[group_id].prob` | `graphrag_prior_prob * evidence_score` | `graphrag_prior_prob` = `mapping.prob` — **null 가능**(오버레이 미매칭 가설). null이면 prior를 패턴 내 균등분배 또는 `occurrence_prior`(high=0.5/mid=0.3/low=0.2 등) 대체 규칙 필요 — §6 L-2 |
| `evidence.drift_detected = None` 조건 | "후보의 telemetry_signature.param == \"none\"" | **`tier != 자동`** (= `verification.fab_table != 'telemetry'`). T5 호출 인자는 `path.evidence` + `verification.direction` |
| (없음) | — | **`scenario_hint`로 A2/A3/A5/A6 체인 분기** — ④가 원인 유형을 자체 판단할 필요 없음 |
| (없음) | — | `verification.unverifiable_signals` / `mapping.param_in_fab_vocab=false` → evidence table **"부족한 데이터"란**(C2)에 기록 |

**원인 명칭 주의**: KG의 `path.cause`는 문헌 추출 id(`etching_non_uniformities`)라
`mapping_table.yaml`의 id(`etch_nonuniformity`)와 다르다. 두 체계의 다리는 출력의
`mapping.matched_cause`다 — ⑥ 응답 생성에서 표 기준 명칭이 필요하면 이것을 쓴다.

§3.1의 evidence 구조·스코어링 공식·Critic 판정 규칙은 v1.0 그대로 유효하다
(단 `prob` null 처리와 `drift_detected` 조건만 위 표대로 교체).

## 5. 저장소 역할 요약 — **[v1.0 대비 Neo4j 행 변경]**

| 저장소 | 성격 | 쓰기 주체 |
| --- | --- | --- |
| fab.db (SQLite) | 정적, read-only, "이번 사실"(운영 데이터) | 시뮬레이터(빌드 타임)만 |
| Neo4j | **준정적 — 문헌 추가 시 재적재됨.** "일반 지식"(문헌 추출 인과 그래프) | **KG 파이프라인(0~5번)**. 서비스는 읽지 않음(아래 파일로 소비) |
| `outputs/hypotheses.json` | Neo4j의 소비용 스냅샷. `meta.generated_at`으로 신선도 검사 | 6번 (`6_ask_graphrag.py`) |
| app_state.db (SQLite) | 매일 갱신, 서비스 운영 상태 | 서비스가 매 배치마다 |

- ⬜ **L-3. ④의 체인 분기를 `scenario_hint`로** — A2/A3/A5/A6 판단 로직을 자체 구현하지 말 것.

- ⬜ **L-4. C2 기록 경로** — `verification.unverifiable_signals`·`mapping.param_in_fab_vocab=false`를
  evidence table "부족한 데이터"란으로 (MCP 문서 C2 규칙)

- ⬜ **L-6. ⑥ 인용 소스 변경** — v1.0 `Citation` 노드는 없음. 인용은 `provenance.quotes`(문헌 원문)
