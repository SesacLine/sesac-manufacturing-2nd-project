# **데이터모델설계**

데이터 저장소는 3종류다 — ① fab.db(SQLite) ② Neo4j(그래프 DB) ③ app_state.db(SQLite) + LangGraph `RCAState`(파이프라인 실행 중 메모리 상의 상태). 이 문서는 이 넷을 한 곳에 모은 것이다.

---

## **1. fab.db (SQLite)**

`metric_series.scope`는 장비 단위이지 로트 단위가 아니다. 로트별 수율(저수율 선별용)은 이 테이블이 아니라 `wafer.die_map`을 직접 집계해서 계산한다.

## **2. Neo4j 그래프 스키마**

패턴(결함 유형) → 원인 지식을 그래프 노드·관계로 변환한 설계다. 구조: "패턴 노드 → cause 노드 → (process 노드, telemetry_signature 노드, citation 노드)".

### **노드 라벨**

| 라벨 | 속성 | 비고 |
| --- | --- | --- |
| `Pattern` | `name`(예: "Edge-Ring") | 3개 인스턴스(Center/Edge-Ring/Scratch) |
| `Cause` | `name`(예: "etch_nonuniformity") | 8개 인스턴스 |
| `Process` | `name`(예: "ETCH") | 공정 단계이자 장비군 |
| `TelemetrySignature` | `param`, `drift` | 시그니처 없는 원인은 관계 생략 |
| `Citation` | `text` | 문헌 인용 텍스트 |

### **관계 타입**

| 관계 | 방향 | 속성 |
| --- | --- | --- |
| `CAUSED_BY` | `(Pattern)-[:CAUSED_BY]->(Cause)` | `prob`(REAL) |
| `IN_PROCESS` | `(Cause)-[:IN_PROCESS]->(Process)` | - |
| `HAS_SIGNATURE` | `(Cause)-[:HAS_SIGNATURE]->(TelemetrySignature)` | - (signature 없는 cause는 생략) |
| `CITED_BY` | `(Cause)-[:CITED_BY]->(Citation)` | - |

### **예시 (Edge-Ring 브랜치, Cypher)**

```
MERGE (p:Pattern {name:"Edge-Ring"})
MERGE (c1:Cause {name:"etch_nonuniformity"})
MERGE (proc1:Process {name:"ETCH"})
MERGE (sig1:TelemetrySignature {param:"rf_power", drift:"step_up"})
MERGE (cit1:Citation {text:"Wang et al., IEEE TSM 2020"})
MERGE (p)-[:CAUSED_BY {prob:0.60}]->(c1)
MERGE (c1)-[:IN_PROCESS]->(proc1)
MERGE (c1)-[:HAS_SIGNATURE]->(sig1)
MERGE (c1)-[:CITED_BY]->(cit1)
```

### **런타임 조회 쿼리**

```
MATCH (p:Pattern {name: $pattern})-[r:CAUSED_BY]->(c:Cause)
OPTIONAL MATCH (c)-[:IN_PROCESS]->(proc:Process)
OPTIONAL MATCH (c)-[:HAS_SIGNATURE]->(sig:TelemetrySignature)
OPTIONAL MATCH (c)-[:CITED_BY]->(cit:Citation)
RETURN c.name AS cause, proc.name AS process, r.prob AS prob, sig, cit.text AS citation
ORDER BY r.prob DESC
```

이 쿼리는 코드에 고정으로 박아두는 파라미터 쿼리다(패턴명 `$pattern`만 바뀜) — 매 요청마다 LLM이 새로 생성하지 않는다.

### **시딩(초기 적재)**

원인 매핑 지식(패턴→원인 정의)을 위 Cypher `MERGE` 문으로 변환하는 1회성 스크립트가 필요하다(런타임 아님, 셋업 단계).

### **향후 다중 홉 질의 확장 지점**

지금 조회는 전부 단일 패턴 키 조회다. "이 원인이 다른 패턴에도 걸쳐 있는가"(예: CMP가 여러 패턴의 원인에 공통으로 등장) 같은 교차 질의를 추가하고 싶다면:

```
MATCH (proc:Process {name:"CMP"})<-[:IN_PROCESS]-(c:Cause)<-[:CAUSED_BY]-(p:Pattern)
RETURN p.name, c.name
```

## **3. LangGraph `RCAState`**

| 필드 | 타입 | 쓰는 노드 | 읽는 노드 |
| --- | --- | --- | --- |
| `cursor_date` | `str` | (서비스가 별도 관리, state 밖에서 주입) | ⓪ |
| `target_lot_ids` | `List[str]` | ⓪ | ① |
| `vlm_results` | `List[{lot_id, wafer_id, pattern, spatial, description, severity, confidence, ambiguity}]` | ① | ② |
| `groups` | `List[{group_id, pattern, lot_ids, status}]` | ② | ③(pattern만), ④(lot_ids만) |
| `graphrag_candidates[group_id]` | `{pattern, candidates:[...]}` | ③ | ④ |
| `hypotheses[group_id]` | `List[{cause, equipment, prob, evidence}]` | ④ | ⑤ |
| `critic_result[group_id]` | `{accepted, rejected, status?}` | ⑤ | ⑥ |
| `final_response[group_id]` | `{group, hypotheses, rejected}` | ⑥ | (프론트엔드로 전달) |

**그룹 팬아웃**: `groups`가 N개일 때 ③~⑤를 그룹별로 순차 루프 돌릴지, LangGraph `Send` API로 병렬 분기할지는 규모에 따라 결정한다 — 지금 규모(하루 그룹 수 적음)면 순차 루프로 충분하다.

### **3.1 ④⑤ 노드화(2026-07-09) — `evidence` 스키마와 스코어링 공식**

Hypothesis·Critic은 LLM 에이전트가 아니라 **결정적 함수(룰베이스) 노드**로 구현한다(근거: `personalspace/0709 work/qna_0709.md` Q6). 이에 맞춰 `evidence` 필드는 "Critic이 실제로 비교하는 구조화된 값"과 "⑥ 응답생성이 나중에 인용할 표시용 문자열"을 분리해서 설계한다 — Critic은 후자를 읽지 않는다.

```python
evidence = {
    # Critic이 실제로 비교하는 구조화된 값
    "commonality_ratio": 0.875,        # float, run_commonality_analysis 결과
    "drift_detected": True,             # bool | None (None = 후보의 telemetry_signature.param == "none")
    "maintenance_hit": True,            # bool, t0 근접 BM/PM 존재 여부
    "maintenance_ts": "2026-01-29 05:58",
    "alarm_hit": True,                  # bool
    "normal_ratio": 0.12,               # float, get_normal_lot_ratio 결과

    # ⑥ 응답생성 전용 표시 문자열(f-string 조합, LLM 생성 아님) — Critic은 미참조
    "telemetry_summary": "rf_power step_up (t0=2026-01-29 05:58)",
    "maintenance_summary": "BM @ t0",
}
```

**Hypothesis의 확률 산출 공식(초기 제안값 — 데모 검증 후 조정)**:

```
evidence_score = mean(
    commonality_ratio,
    1 if drift_detected else 0,       # drift_detected가 None이면 이 항목 제외하고 나머지로 평균
    1 if maintenance_hit else 0,
    1 if alarm_hit else 0,
    1 - normal_ratio,
)
prob = graphrag_prior_prob * evidence_score     # GraphRAG 사전확률에 증거강도를 곱해 보정
```

**Critic의 판정 규칙**:

- 시간정합성: `maintenance_ts > 결함_ts` → 기각(트랩)
- 반대증거: `normal_ratio > NORMAL_RATIO_THRESHOLD`(초기값 제안: 0.5) → 기각
- 위 두 조건을 통과한 후보만 채택. **채택 후보가 0개면 재시도 없이 즉시 `critic_result.status = "insufficient_evidence"`** 반환(과거의 "재계획 루프"를 대체).

## **4. app_state.db (SQLite)**

fab.db는 read-only라 "오늘 날짜 커서"와 "배치 결과"는 별도 파일에 저장한다.

```sql
-- 오늘 날짜 커서 (단일 행)
CREATE TABLE cursor_state (
    id INTEGER PRIMARY KEY CHECK (id = 1),
    current_date TEXT NOT NULL
);

-- 그룹별 배치 실행 결과 (대시보드 큐 조회용)
CREATE TABLE batch_group_result (
    group_id TEXT PRIMARY KEY,
    run_date TEXT NOT NULL,
    pattern TEXT NOT NULL,
    lot_ids TEXT NOT NULL,        -- JSON 배열
    status TEXT NOT NULL,          -- ready_for_commonality / insufficient_evidence / no_mapping / done
    final_response TEXT,           -- JSON (⑥ 응답생성 결과 그대로)
    created_at TEXT NOT NULL
);
```

LangGraph의 `SqliteSaver` 체크포인터는 "파이프라인 실행 도중의 중간 상태"(노드화 이후엔 재시도·재계획 루프가 없으므로, ⓪~⑥ 각 노드의 순차 실행 진행 상황·중단 시 재개 지점을 저장하는 용도로 성격이 바뀜)를 자동 저장하고, 위 `batch_group_result`는 "대시보드가 완료된 결과를 빠르게 조회"하기 위한 별도 조회용 테이블이다 — 둘은 목적이 달라 같이 쓴다.

## **5. 세 저장소의 역할 요약**

| 저장소 | 성격 | 쓰기 주체 |
| --- | --- | --- |
| fab.db (SQLite) | 정적, read-only, "이번 사실"(운영 데이터) | 시뮬레이터(빌드 타임)만, 서비스는 읽기 전용 |
| Neo4j | 정적에 가까움, "일반 지식"(문헌 기반 원인후보) | 시딩 스크립트(셋업 1회), 서비스는 읽기 전용 |
| app_state.db (SQLite) | 매일 갱신, 서비스 자체 운영 상태 | 서비스가 매 배치 실행마다 씀 |
